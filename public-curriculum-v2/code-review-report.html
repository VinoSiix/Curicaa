🛡 Security

Authentication vulnerabilities, data exposure risks, and missing protections. These should be addressed before any production deployment.

Passwords stored in plain text
Critical
User passwords are stored as raw strings in localStorage. Anyone with browser access (dev tools, XSS, shared device) can read every user's password. This is a fundamental security violation.
File: auth.js line 45
Current (vulnerable)
var user = { id: Date.now().toString(36) + Math.random().toString(36).slice(2, 7), name: name.trim(), email: email.trim().toLowerCase(), password: password, // Plain text! plan: 'free', createdAt: new Date().toISOString(), grades: [] };
Fix
Hash passwords with a library like bcrypt.js before storage. For a client-side app, use Web Crypto API's crypto.subtle.digest('SHA-256', ...) with a per-user salt. Even better: move authentication to a real backend.
No input sanitization on user data
Critical
User-supplied strings (name, email) are stored and later rendered via textContent in some places, but innerHTML is used elsewhere (e.g., plan badge updates). This creates stored XSS vectors where a malicious name like <img onerror=alert(1)> could execute arbitrary JavaScript.
Files: auth.js lines 34-53, hub-premium.js lines 307-332, 507
Current (XSS risk)
// hub-premium.js line 507 — innerHTML with user-controlled badge text badge.innerHTML = '<i class="fas fa-circle"></i> ' + planLabel;
Fix
Sanitize all user input before storage using a DOM-based sanitizer. Never use innerHTML with user data — always use textContent. The settings modal already does this correctly (lines 472-498); apply the same pattern everywhere.
localStorage used as primary database
High
All user accounts, sessions, and plan data live in localStorage. Clearing browser data destroys all accounts. localStorage is limited to ~5MB, has no query capability, and is accessible to any script on the same origin. This architecture cannot scale past a demo.
File: auth.js lines 7-27
var DB_KEY = 'curicaa_users'; var SESSION_KEY = 'curicaa_session'; function getUsers() { try { return JSON.parse(localStorage.getItem(DB_KEY)) || []; } catch(e) { return []; } } function saveUsers(users) { localStorage.setItem(DB_KEY, JSON.stringify(users)); }
Fix
Implement a real backend (Node/Express, Firebase Auth, or Supabase). For a quick migration path, use Firebase Auth for authentication and Firestore for user data. This also enables multi-device sessions and proper security rules.
Demo checkout with fake card input
High
The checkout flow accepts fake card numbers and processes them as if real. While this may be intentional for demo purposes, there is no clear indication to users that no real payment is happening. Users may enter real card details expecting a transaction.
File: hub.html ~line 521 (checkout modal)
Fix
Add a visible "Demo Mode" banner to the checkout modal. When real payments are ready, integrate Stripe Elements or a similar PCI-compliant payment provider. Never accept raw card numbers in your own form fields.
Missing CSRF protection
Medium
There are no CSRF tokens or SameSite cookie attributes protecting state-changing operations. While the current localStorage architecture is inherently immune to traditional CSRF, migrating to a backend without adding protection would be dangerous.
Files: auth.js, hub-premium.js (all mutation functions)
Fix
When migrating to a backend: use SameSite=Strict cookies for session tokens, implement CSRF token validation on all state-changing endpoints, and use the Origin header check as a secondary defense.
No rate limiting on login
Medium
The login function has no brute-force protection. An attacker could attempt unlimited password guesses per second. Combined with plain-text password storage, this means any account can be compromised through dictionary attacks.
File: auth.js lines 56-64
login: function(email, password) { var users = getUsers(); var user = users.find(function(u) { return u.email === email && u.password === password; }); // No attempt counting, no delay, no lockout }
Fix
Add exponential backoff on failed attempts (store attempt count + timestamp in localStorage for now). When moving to a backend, implement proper rate limiting (e.g., express-rate-limit) and account lockout after N failures.
Weak email validation
Low
The email check is !email.includes('@'), which accepts strings like @, @@, or a@ as valid emails. This provides essentially no validation.
File: auth.js line 39
Current
if (!email || !email.includes('@')) return { ok: false, error: 'Please enter a valid email.' };
Recommended
var emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/; if (!email || !emailRegex.test(email)) return { ok: false, error: 'Please enter a valid email.' };

⚡ Performance

Render-blocking resources, heavy DOM manipulation, and optimization opportunities that affect page load and runtime performance.

Massive inline CSS block (~11KB)
High
The entire stylesheet is embedded in the HTML <style> tag (lines 13-140), weighing approximately 11KB. This blocks HTML parsing, prevents caching, and means the full CSS is re-downloaded on every page visit. Combined with 200+ inline style="" attributes, the styling architecture has no separation of concerns.
File: hub.html lines 13-140
Fix
Extract CSS to an external file and load with <link rel="stylesheet">. Move inline styles to CSS classes. This enables browser caching and reduces HTML payload by ~15%.
90 animated DOM star elements
High
The page creates 90 <div class="star"> elements, each running a CSS twinkle animation with unique durations and delays. This creates 90 concurrent animation loops and 90 active layout participants. On lower-end devices, this causes noticeable jank.
File: hub-premium.js (star generation in init)
Fix
Replace DOM-based stars with a single <canvas> element. A canvas-based star field can render hundreds of stars at 60fps with minimal CPU cost. Alternatively, use CSS will-change: opacity and reduce to ~30 stars.
No lazy loading for images or off-screen content
Medium
All content renders immediately on page load. Sections below the fold (testimonials, pricing, FAQ, footer) could be deferred. No images use loading="lazy" or IntersectionObserver-based loading.
File: hub.html (all <img> tags)
Fix
Add loading="lazy" to all below-fold images. Use IntersectionObserver to defer rendering of heavy sections (like the testimonial carousel) until they approach the viewport.
Font Awesome loaded synchronously
Medium
Font Awesome CSS is loaded via a synchronous <link> tag in the <head>. The browser must download and parse this file before rendering any content. For an icon font with hundreds of glyphs (most unused), this is wasteful.
File: hub.html line 7
<link rel="stylesheet" href="assets/font-awesome.css">
Fix
Load Font Awesome asynchronously with media="print" onload="this.media='all'", or switch to an SVG icon system that only bundles the ~20 icons actually used. Alternatively, use rel="preload" to avoid render blocking.

The Google Fonts URL already uses display=swap, which is good. For extra resilience, add a @font-face declaration with font-display: swap as a local fallback.

♿ Accessibility

WCAG compliance gaps, keyboard navigation issues, and assistive technology support. These affect users with disabilities and can have legal implications.

Interactive elements are <div>s and <span>s
High
Many clickable elements (FAQ items, carousel arrows, modal triggers) use <div> or <span> instead of <button> or <a>. While some have tabindex="0" and role="button", they lack proper keyboard activation (Enter/Space), screen reader announcements, and focus styling.
Files: hub.html lines 148, 218, hub-premium.js line 166
Fix
Replace all interactive <div>s with semantic <button> elements. Buttons get keyboard support, focus styling, and screen reader semantics for free. The pathway cards already use role="button" — convert them to actual <button>s.
Modal focus trap missing
High
When a modal opens, focus is not trapped within it. Tab key moves focus to elements behind the modal overlay. Screen reader users can navigate away from the modal without closing it, creating a confusing experience.
Files: hub.html (all 6 modals), hub-premium.js lines 302-305
Fix
Implement a focus trap: when the modal opens, move focus to the first focusable element. On Tab/Shift+Tab, cycle focus within the modal. On Escape, close the modal and restore focus to the trigger element. Use the native <dialog> element for built-in focus trapping.
Missing aria-expanded on FAQ items
Medium
FAQ items toggle open/closed via JavaScript, but they lack aria-expanded attributes. Screen readers cannot determine whether a FAQ answer is currently visible or hidden.

el.setAttribute('aria-expanded', 'false'); el.setAttribute('role', 'button'); // On toggle: el.setAttribute('aria-expanded', isOpen ? 'true' : 'false');
Form inputs lack associated labels

Medium
Several text color values are very low opacity white against a near-black background. For example, --text-muted: rgba(255,255,255,0.45) on #0f1119 yields a contrast ratio of approximately 3.2:1, below the WCAG AA minimum of 4.5:1 for normal text.

💻 Code Quality

Maintainability concerns, patterns that increase technical debt, and structural improvements for long-term code health.

Mixed var / let / const usage
Medium
The codebase uses var exclusively in hub-premium.js and auth.js, while hub.html has no JavaScript variable declarations. All var declarations should be migrated to const (for values that never change) and let (for values that do). var has function-scoping and hoisting behaviors that cause subtle bugs.
Files: hub-premium.js (throughout), auth.js (throughout)
Fix
Replace all var with const by default. Only use let for variables that are reassigned (like loop counters or toggles). This can be done with an automated ESLint fix: --fix with the prefer-const rule.
Global function pollution
Medium
Over 40 functions are defined in the global scope (handleLogin, openLoginModal, toggleFaq, launchConfetti, etc.). This pollutes the global namespace, risks naming collisions with third-party scripts, and makes dependency tracking impossible.
File: hub-premium.js (throughout)
Fix
Wrap all hub-premium.js code in an IIFE (like auth.js already does). Export only the functions that HTML onclick handlers need via a namespace object: window.CuricaaHub = { handleLogin, openLoginModal, ... }.
Duplicate plan price logic
Medium
Plan prices are defined in multiple places: once in the HTML for the pricing cards, and again in JavaScript for the upgrade/toggle logic. When prices change, they must be updated in both places, creating a synchronization risk.

Define all pricing data in a single source of truth (e.g., a PRICING object in JavaScript). Render pricing cards dynamically from this data, just like testimonials and FAQ are already rendered.

🚀 Improvements

15 actionable enhancements that would significantly improve the codebase, user experience, and scalability.

Large Effort

1. Add a build step (Vite or Parcel)

A build tool enables minification, tree-shaking, CSS extraction, image optimization, and hot module replacement during development. This single change unlocks most other performance improvements. Vite is recommended for its fast dev server and simple config.

Large Effort

2. Convert to a component system

Break the monolithic HTML into reusable components (header, pricing card, FAQ item, modal). This can be done with vanilla JS web components, or a lightweight framework like Preact. The current render functions (renderFAQ, renderTestimonials) are already halfway there.

Large Effort

3. Add real payment processing (Stripe)

Integrate Stripe Checkout or Stripe Elements for PCI-compliant payment handling. This replaces the demo checkout flow and enables real revenue. Stripe also handles subscriptions, invoicing, and refund management.

Medium Effort

4. Implement a service worker

Since the curriculum already "works offline," complete the offline story with a service worker. Cache static assets for instant repeat loads and implement offline fallback pages. Use Workbox for easy implementation.

Small Effort

5. Add privacy-first analytics

Use Plausible Analytics or Umami (self-hosted) for GDPR-compliant, cookie-free analytics. Understand user behavior without compromising privacy. Avoid Google Analytics for a family-focused educational product.

Medium Effort

6. Add E2E tests (Playwright)

Write end-to-end tests for critical flows: signup, login, purchase, plan upgrade, navigation. Playwright is recommended since the page is pure HTML/JS. Start with smoke tests for each modal and the checkout flow.

Small Effort

7. Optimize CSS custom properties

Consolidate redundant variables (--bg-subtle vs --bg-card), remove unused ones, and document the remaining design tokens in a style guide. This reduces CSS size and improves developer experience.

Small Effort

8. Add keyboard shortcuts

Support Escape to close modals (already partially done), ? for help, and / for search. Power users and accessibility-focused users benefit greatly from keyboard navigation.

Medium Effort

9. Implement real search functionality

Allow users to search across grades, subjects, and lessons. A client-side search using Fuse.js or FlexSearch would work with the existing static content and add significant navigational value.

Small Effort

10. Add progress persistence

Track which lessons a user has completed and display progress on the pathway cards. Store progress in localStorage (for now) and sync to a backend later. Visual progress indicators increase engagement.

Small Effort

11. Use Inter font subset

The Google Fonts URL loads the entire Inter variable font (~300KB). Use the Unicode-range subsetting to load only Latin characters, or host a self-hosted WOFF2 subset. This reduces font payload by 50-70%.

Small Effort

12. Add a print stylesheet

Parents may want to print lesson plans. A @media print stylesheet should hide navigation, modals, and decorative elements, then format the lesson content for paper. Add page-break-after rules for clean pagination.

Small Effort

13. Add rate limiting for lead capture

If the contact/lead capture forms are added, implement client-side rate limiting to prevent spam. Server-side validation is essential when a backend exists. Use a honeypot field and time-based throttling.

Small Effort

14. Implement toast notifications

Replace the error message divs in modals with a global toast notification system. Toasts provide consistent, non-intrusive feedback for all user actions (login success, plan upgraded, settings saved). They auto-dismiss and stack gracefully.

Large Effort

15. Create a design system

Document the color palette, typography scale, spacing system, component patterns (cards, buttons, modals), and animation curves in a shared reference. This ensures consistency as the team grows and makes onboarding faster. Use Storybook for interactive documentation.

The Curicaa Hub is a visually polished, well-designed landing page with a strong user-facing experience. The dark theme, animations, and micro-interactions show clear attention to design detail. The semantic HTML foundation (skip link, landmarks, proper headings) is a good starting point for accessibility.


However, the codebase has significant architectural concerns that would block a production launch. The security model (plain-text passwords in localStorage, no sanitization) is the most urgent area. The code organization (200+ inline styles, global functions, no build step) creates a maintenance burden that will compound as features are added.


The recommended path forward: fix the 3 security blockers first, then establish a build step and extract CSS. These two changes unlock the rest of the improvements and transform the codebase from a prototype into a production-ready product.
