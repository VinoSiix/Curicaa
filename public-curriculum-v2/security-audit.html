Critical
Plan Escalation via Client-Side updatePlan() — FIXED auth.js:251 · hub-premium.js:1225
▼

Description

The handlePurchase() function in hub-premium.js directly calls CuricaaAuth.updatePlan(newPlan, mergedGrades) which writes the new plan to the profiles table in Supabase. Any logged-in user could call this function from the browser console to escalate their plan from "free" to "pro" without paying.

Exploit PoC

// Open browser console on hub.html while logged in: CuricaaAuth.updatePlan('pro', []) // Profile now reads "pro" — all premium content unlocked

Mitigation — RLS Policy (Your Current Defense)

Your supabase-schema.sql line 42-44 includes a WITH CHECK constraint that prevents plan changes:

WITH CHECK ( auth.uid() = id AND (plan, role) = ( SELECT plan, role FROM profiles WHERE id = auth.uid() ) ) -- This means: a user can only update their profile if the NEW plan -- equals the OLD plan. Plan escalation is BLOCKED at the database level.

Verdict: The RLS policy should prevent this. You MUST verify this policy is active in your Supabase dashboard before launch. If the policy is missing or was dropped during a migration, this becomes an actual exploitable vulnerability.

Recommended Fix (Pre-Stripe)

  • Verify RLS WITH CHECK policy is active on the profiles table in Supabase
  • Remove updatePlan from the client-side API surface entirely — plan changes should only happen server-side via Stripe webhooks
  • Add a Supabase Edge Function or RPC that validates a Stripe session before updating the plan

Admin Dashboard Anon Read Access — FIXED supabase-schema.sql:30-33, 117-121
▼

Description

The original schema included policies allowing anon role to SELECT all rows from profiles and waitlist tables. This means any unauthenticated user could read all user emails, names, and plan information using the Supabase REST API with the anon key.

Current Status

Your schema file (lines 258-261) includes migration statements to DROP these policies:

DROP POLICY IF EXISTS "Allow admin dashboard read on profiles" ON profiles; DROP POLICY IF EXISTS "Allow admin dashboard read on waitlist" ON waitlist;

You MUST verify these DROP POLICY statements have been executed in production. If they haven't, all user PII (names, emails, plans) is publicly readable via the Supabase REST API.

How to Verify

// Run in Supabase SQL Editor: SELECT policyname, tablename FROM pg_policies WHERE schemaname = 'public' AND policyname LIKE '%admin dashboard%'; // Should return 0 rows. If it returns rows, the policies are still active.

Description

Missing HSTS Header — FIXED vercel.json
▼

Description

While Vercel provides HTTPS by default, the Strict-Transport-Security header is not configured. Without HSTS, a user's first visit via HTTP could be intercepted via SSL stripping before being redirected to HTTPS.

Fix

Add to the headers array in vercel.json:

{ "key": "Strict-Transport-Security", "value": "max-age=63072000; includeSubDomains; preload" }

▼

Description

Supabase stores JWT tokens in localStorage by default. This makes them accessible to any JavaScript running on the page, including XSS payloads or malicious browser extensions.

Risk

If an XSS vulnerability were discovered, the attacker could steal auth tokens directly from localStorage. However, the current codebase properly sanitizes user input via the esc() function.

  • For the current static site, localStorage is acceptable given the thorough XSS prevention

Medium
CSP Allows 'unsafe-inline' — DEFERRED (requires refactor) vercel.json:23
▼

Description

The CSP header includes 'unsafe-inline' for both script-src and style-src. This significantly weakens XSS protection because inline scripts and event handlers (onclick, etc.) are allowed to execute.

Why It Exists

The entire site uses inline event handlers (onclick="...") and inline styles. Removing unsafe-inline would require a major refactor to move all handlers to addEventListener calls and all styles to CSS files.

Recommended Fix

  • Long-term: Migrate all inline handlers to external JS with addEventListener
  • Add a nonce-based CSP: script-src 'self' 'nonce-{RANDOM}'
  • Short-term: Accept as-is, since the esc() function provides XSS defense-in-depth
